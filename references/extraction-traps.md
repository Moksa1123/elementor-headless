# Eleven traps that make an Elementor schema silently wrong

Everything in `data/` is generated by `tools/extract-elementor-schema.php` running
against a live install. That sounds simple, and the naive version of it is wrong
in eleven separate ways — each of which produces a schema that *looks* complete,
passes a casual eyeball check, and quietly lies to you.

All eleven were found the hard way, in this repo, after shipping data that had them.
Some came out of reading Elementor's source. The rest only surfaced when every
control in the schema was rendered and checked, one at a time — and those could
*only* be found that way, because they are Elementor's own metadata being wrong,
or a whole output channel the verifier was not looking at.

The order below is the order they were found, which is also the order of how hard
they are to see:

| # | Trap | Found by |
|---|---|---|
| 1 | WP-CLI gets a stripped control stack | reading the source |
| 2 | responsive is two mechanisms, not one | reading the source |
| 3 | a control's tier is not its widget's tier | measuring Free vs Pro |
| 4 | a control is gated three different ways | [the CSS sweep](#the-sweep-render-every-control-and-look) |
| 5 | responsive dependencies are re-checked at the breakpoint | the CSS sweep |
| 6 | `is_responsive` over-promises | the CSS sweep |
| 7 | **CSS is only half of what a control can do** | [the HTML sweep](#trap-7--css-is-only-half-of-what-a-control-can-do) |
| 8 | **a class control's value is remapped, and its prefix changes per device** | the HTML sweep |
| 9 | **writing `_elementor_data` leaves a stale rendered-HTML cache** | the HTML sweep |
| 10 | **the widget surface is a property of the INSTALL, not of Elementor** | [extracting from a second site](#trap-10--the-surface-is-a-property-of-the-install-not-of-elementor) |
| 11 | **a rule can be in the stylesheet and LOSE** | [a real browser](#trap-11--a-rule-can-be-in-the-stylesheet-and-lose) |

Traps 7-9 are the ones a stylesheet-based verifier is structurally incapable of
finding, however green it runs. Trap 9 in particular makes a *correct* tree render
as the *previous* page, with no error anywhere — and this repo shipped it under a
sweep that reported 0 failures across 17,421 controls.

Trap 10 is the one no amount of verification on a single machine can find, because
every check agrees with every other check — they are all looking at the same site.

## Trap 1 — WP-CLI looks like the front end, so Elementor hands you a stripped stack

`Elementor\Core\Frontend\Performance::should_optimize_controls()`:

```php
public static function should_optimize_controls() {
    if ( null === static::$is_frontend ) {
        static::$is_frontend = (
            ! is_admin()
            && ! Plugin::$instance->preview->is_preview_mode()
            && ! defined( 'REST_REQUEST' )
        );
    }
    return static::$is_frontend;
}
```

A `wp eval-file` run is not admin, not preview, and not REST. All three
conditions pass, so Elementor concludes it is rendering the front end and
switches to its lean control path — the one built for speed, not for
introspection. Two things then happen:

| Where | What it does |
|---|---|
| `includes/base/controls-stack.php` | UI controls (heading, divider, raw_html, button, alert, notice) are cut down to `['type', 'section']` — labels gone |
| `includes/managers/controls.php` | Every control with `selectors` (i.e. every *style* control) is diverted into a separate `style_controls` stack, and section markers lose their `label` and their real `tab`, defaulting to `'content'` |

Measured on Elementor 4.1.4:

```
                        optimised    full     lost
container controls            310     458    32%
all settable controls      20,485  37,964    46%
controls carrying a label       7  36,753    ~100%
tab attribution           all 'content'  layout/style/advanced   broken
```

**Nearly half the control surface, and every scrap of tab and label metadata,
vanishes — with no error.** A schema built this way will tell you the Heading
widget has no style controls.

The fix is to turn the flag off before anything touches an element:

```php
$rp = new ReflectionProperty( \Elementor\Core\Frontend\Performance::class, 'is_frontend' );
$rp->setAccessible( true );
$rp->setValue( null, false );
```

Order matters: `$is_frontend` is a cached static, and control stacks are built
lazily on first `get_controls()`. Flip it first, before you read anything.

Two consequences to know about:

- With optimisation off, Elementor `wp_die()`s on any addon that registers a
  control outside a section (`controls-stack.php` only tolerates that in the
  optimised path). One sloppy third-party plugin kills the whole extraction. The
  extractor therefore decides a widget's tier *before* calling `get_controls()`
  on it, and never builds controls for widgets it is going to discard.
- Labels are translated at registration time, so the dump inherits the site's
  locale. `switch_to_locale( 'en_US' )` before extracting, or you ship a schema
  full of `'背景'` where the world expects `'Background'`.

The extractor asserts its way out of this rather than trusting it worked: it
checks that `container` / `background_color` reports `tab = 'style'`, and exits
non-zero if not.

## Trap 2 — responsive is two mechanisms, and the obvious test only finds one

Half the point of the schema is answering "can I write `padding_tablet` here?".
Getting that wrong in the *rejecting* direction is worse than not knowing.

`add_responsive_control()` does not do one thing. It does two, and they look
nothing alike in the control stack:

**Mechanism A — the common one.** The control is registered **once**. There is no
`padding_tablet` entry in the stack at all; the breakpoint variants are resolved
at render time by looking up `"{control}_{device}"` in the saved settings. The
raw args of the container's `padding`:

```php
'type'          => 'dimensions',
'responsive'    => array(),      // EMPTY
'is_responsive' => true,         // <- the actual flag
```

**Mechanism B.** Real suffixed siblings exist in the stack alongside the base:

```php
// sticky_offset
'responsive' => array( 'max' => 'desktop' ),   // non-empty
// and sticky_offset_tablet, sticky_offset_mobile are separate controls
```

Testing `! empty( $ctrl['responsive'] )` — the obvious test — catches **only
mechanism B**. On Elementor 4.1.4 that is 58 controls on the container while
mechanism A covers the ones everybody actually uses: padding, margin, width,
font size, gap.

Before the fix, 9.8% of controls were marked responsive. After: **30.1%**. The
schema had been rejecting `padding_tablet` as an unknown control.

The authoritative flag is **`is_responsive`**. A control is responsive if
`is_responsive` is true *or* it has suffixed siblings; the usable suffixes are
the site's **active** breakpoints (`el.py breakpoints`).

The extractor has a second canary for this: if `container` / `padding` does not
come out responsive, it refuses to emit.

**This is also why `verify-render.py` exists.** A control name you can look up in
a stack can be checked against that stack. A settings key with *no control object
behind it anywhere* cannot — the only way to know `padding_tablet` works is to
write it, let Elementor compile the CSS, and confirm a `@media(max-width:1024px)`
rule came out the other end. Which it does:

```css
@media(max-width:1024px){ .elementor-element-1a2b3c4{ --padding-top:56px; ... } }
```

## Trap 3 — a control's tier is not its widget's tier

You can read a *widget's* tier off the filesystem: whichever plugin directory
defines the class. You cannot do that for a *control*, because **Elementor Pro
reaches into the free widgets and injects controls into them.**

Open the free Heading widget on a site with Pro and you will find Motion Effects,
Sticky, Custom CSS, Display Conditions and Custom Attributes sitting in its
Advanced tab. Inherit the owner's tier and all of them get labelled `free`. Build
on that assumption and the page renders perfectly on your machine and silently
loses its styling on a Free install. No error. No warning.

Measuring it is easy once you stop trying to reason about it:

```bash
wp eval-file tools/extract-elementor-schema.php core+pro > pro.json
wp --skip-plugins=elementor-pro eval-file tools/extract-elementor-schema.php core+pro > free.json
python tools/build-indexes.py pro.json --free-dump free.json --out data/
```

`--skip-plugins` affects **only that one CLI process**. Nothing is deactivated;
this is safe to run against production.

Diff the two and the boundary falls out — measured, not argued:

| | Free 4.1.4 | + Pro 4.1.2 |
|---|---|---|
| controls on every widget | 165 | **211** (+46) |
| controls on `container` | 277 | **356** (+79) |
| control types | 52 | **59** (+8, −1) |
| group controls | 11 | **16** (+5) |

The 46 injected into **every** widget:

| Section | n | What |
|---|---|---|
| `section_effects` | 43 | all `motion_fx_*` (37) + `sticky*` (6) |
| `section_custom_css` | 1 | `custom_css` |
| `_section_attributes` | 1 | `_attributes` — custom HTML attributes. **Pro.** Most people assume free |
| `_section_style` | 1 | `e_display_conditions` |

Pro-only group controls: `motion_fx`, `posts`, `query-group`, `related-query`,
`taxonomy-query`.
Pro-only control types: `query`, `template_query`, `conditions_repeater`,
`form-fields-repeater`, `fields_map`, `media-preview`,
`animation_menu_dropdown`, `animation_slides_content`.

### The counting quirk you will hit, and what it means

The Pro-less build reports **more** widgets than the Pro build (145 vs 135), and
the extras are all WooCommerce ones. That is not a bug in the diff.

When Pro is inactive, **Elementor core registers stub "upgrade to Pro" widgets**
so the editor can advertise them. They carry exactly the shared common controls
and nothing of their own, and they are identifiable by a control type that exists
*only* when Pro is absent: **`promotion_control`**
(`Elementor\Modules\Promotions\Controls\Promotion_Control`).

So a widget merely *existing* in the Pro-less build does not make it free. The
tier logic recognises the stubs (zero controls of their own) and does not treat
their controls as free. Once that is accounted for, the books balance exactly:

```
52 free control types  −  1 (promotion_control, free-only)  +  8 (Pro-only)  =  59
```

and the two independent methods — tier-by-filepath and tier-by-presence — agree on
every user-facing widget. The only five they disagree on (`common`, `common-base`,
`common-optimized`, `e-component`, `inner-section`) are Elementor's internal base
classes, which have no controls of their own and are not widgets anyone places.

### One more piece of semantics

A control's tier answers: *"assuming I can use this widget at all, does **this
control** additionally require Pro?"* On a widget that is itself Pro the question
is circular, so per-control tier there carries no information and is not checked.
What matters — and what is checked, on all 15,969 of them — is a Pro-only control
sitting on a **free** widget.

## Trap 4 — a control has THREE ways of being gated, and `condition` is only one

This is the one you cannot find by reading the schema. It only shows up when you
set a control, render the page, and notice nothing happened.

A control emits its CSS only if **all three** of these hold. Handle one or two of
them and a large fraction of Elementor looks broken when it is merely gated.

### 4a. `condition` — the simple form (7,946 controls)

A flat map, checked with equality or membership. Three syntaxes, all of which
appear in real controls:

```jsonc
"background_background": ["classic", "gradient"]   // must be one of
"typography_typography!": ""                       // trailing ! = must NOT equal
"selected_icon[value]!": ""                        // index into the value object
```

A negated condition can carry a **list**, and that list is a set of values to
*avoid* — not a single one:

```jsonc
"_border_color": { "condition": { "_border_border!": ["", "none"] } }
```

`_border_border` must be neither `""` **nor `"none"`**. "Pick the first non-empty
option" lands you on `"none"`, which is precisely the value that switches the
border off — so the colour and the width never emit and the control looks broken
when it is not. This cost 3,123 false failures in the first sweep run.

### 4b. `conditions` — the advanced form (152 controls)

A completely separate mechanism with its own syntax: a boolean tree with `relation`
(and/or), nested `terms`, and comparison operators.

```jsonc
// text-editor.column_gap
{ "relation": "or",
  "terms": [ { "name": "text_columns", "operator": ">",   "value": 1 },
             { "name": "text_columns", "operator": "===", "value": "" } ] }
```

The operators are exactly Elementor's (`includes/conditions.php::compare`):
`==`, `!=`, `!==`, `in`, `!in`, `contains`, `!contains`, `<`, `<=`, `>`, `>=`, and
`===` by default.

**Controls gated this way have an empty `condition`.** Check only `condition` and
they read as unconditional — so you set one, nothing happens, and there is nothing
in the schema to explain why. 152 controls, including `counter.number_gap`,
`off-canvas.offcanvas_animation_duration` and every `motion_fx` transform-origin.

### 4c. `needs_value` — not a condition at all

The killer, because no condition is unmet and nothing looks wrong.

Elementor's CSS generator (`core/files/css/base.php`) expands every `{{...}}`
placeholder in a declaration, and:

```php
if ( '' === $parsed_value ) {
    ...
    throw new \Exception();
}
} catch ( \Exception $e ) {
    return;                 // <- the entire rule is abandoned
}
```

So a declaration that interpolates **another control's** value emits nothing when
that other control is empty. The gradient is the classic:

```
background-image: linear-gradient({{SIZE}}{{UNIT}},
    {{background_color.VALUE}} {{background_color_stop.SIZE}}...,
    {{background_color_b.VALUE}} ...)
```

Set `background_background: "gradient"` and `background_gradient_angle`, satisfy
every documented condition, and you get **no gradient** — because
`background_color` and `background_color_b` are empty. There is no error. There is
nothing in `condition` to warn you.

**499 controls (4.9%) have one of these hidden cross-control dependencies.** The
extractor now parses them out of the `selectors` map and stores them as
`needs_value`; `el.py` prints them as `needs-value-of:`; and `validate-page.py`
makes it an error rather than letting you ship a gradient that does nothing.

## The sweep: render every control and look

`verify-render.py` checks a page you wrote. That only ever covers the controls you
happened to use — 94 of them, on the demo page. It says nothing about the other
16,000.

`tools/sweep-controls.py` closes that gap. For **every** control in the schema that
claims to drive CSS, it synthesises a legal value, solves the dependency chain
needed to make the control take effect (all three mechanisms above), writes the
pages, reads back the stylesheet Elementor compiled, and asserts the value came out.

Each control gets a value **unique to it** — a distinct hex colour, a distinct
pixel size — so "did `title_color` work?" is answered by finding that exact colour
in the CSS, not by noticing that *something* set a `color`. Property-presence would
pass on a page where a different control happened to write the same property.

```bash
python tools/sweep-controls.py plan --out sweep/ --post-id <a draft post>
# apply each batch, capture post-<id>.css  (sweep/RUN.sh shows the loop)
python tools/sweep-controls.py check sweep/ --out data/control-verification.csv
```

Result on Elementor 4.1.4 / Pro 4.1.2 — 550 element variants across 55 pages:

```
DESKTOP  (18,853 CSS-driving controls)
  verified by value   17,421  (92.4%)   the exact value we wrote is in the CSS
  property only        1,270  ( 6.7%)   right property, value not literally assertable
  FAILED                   0  ( 0.0%)
  skipped, untested      162  ( 0.9%)   no test could be built for these
  covered                      99.1%

RESPONSIVE SUFFIXES  (25,404 _tablet / _mobile keys, each asserted inside ITS
                      breakpoint's media query, with a value distinct from desktop's
                      so a leak cannot pass)
  verified by value   24,568  (96.7%)
  property only          819  ( 3.2%)
  FAILED                  17  ( 0.1%)   <- all 9 controls behind them are now
                                           flagged `responsive_broken` in the schema
```

Per-control results ship in **`data/control-verification.csv`**, so "is this control
actually known to work?" is a lookup rather than a hope.

### What the sweep found

The first run failed on **3,123 controls (17.2%)**. Every one of them was a bug —
in the sweep's own solver, or in the schema, or in Elementor:

| Failures | Cause | Whose bug |
|---|---|---|
| ~1,300 | negated conditions with a list (`_border_border!: ['', 'none']`) satisfied with `'none'` | the solver's |
| ~900 | media controls given an empty url, so `url("{{URL}}")` interpolated to nothing and the rule was dropped | the solver's |
| ~800 | gradient/background controls whose CSS interpolates sibling controls that were never set | **the schema's** — it had no `needs_value` |
| 6 | controls gated only by the advanced `conditions` form | **the schema's** — nothing read it |
| 1 | `counter.number_gap` — see below | **the tooling's**, and an Elementor defect |

After fixing all of them: **0 failures**.

### An actual Elementor defect, found by looking

The last failure was `counter.number_gap`. Its conditions were satisfied, its value
was written, and no `gap` appeared. It turned out `gap` **was** emitted — into a
rule that Elementor had already corrupted:

```css
.elementor-element-0994e53 .elementor-counter-number-wrapper{
    text-align:{{VALUE}};        /* a literal, unexpanded placeholder */
    --counter-suffix-grow:1;
    gap:245px;                   /* our value, right there */
}
```

**Elementor 4.1.4 writes a literal `{{VALUE}}` into the compiled stylesheet** on the
`counter` widget. That is invalid CSS, and the stray braces stop any `[^{}]*`-style
parser mid-rule — which is why the sweep reported a false failure that looked
exactly like a schema bug. Both `sweep-controls.py` and `verify-render.py` now
count braces instead.

The lesson generalises past this one widget: **do not parse Elementor's compiled
CSS with a regex that assumes it is well-formed.** It is not, always.

## Trap 5 — a responsive control's dependencies are re-checked AT the breakpoint

This one only exists because Elementor resolves responsive CSS per device, and
almost nobody realises it applies to *dependencies* as well as to the control.

`controls-stack.php`, inside `is_control_visible()`:

```php
$are_control_and_condition_responsive =
    isset( $control['responsive'] ) && ! empty( $controls[ $pure_condition_key ]['responsive'] );

if ( $are_control_and_condition_responsive ) {
    $device_suffix = Controls_Manager::get_responsive_control_device_suffix( $control );
    $condition_name_to_check = $pure_condition_key . $device_suffix;   // <- the TABLET key
}
```

**If a control and its condition dependency are both responsive, the tablet rule's
condition is checked against the dependency's `_tablet` value — not its desktop
one.** Set `grid_rows_grid` (unit `custom`) and `grid_align_content_tablet`, and the
tablet rule never appears: Elementor looked for `grid_rows_grid_tablet`, found
nothing, and hid the control at that breakpoint. Desktop is fine. Tablet is silently
blank.

The same thing happens through the interpolation path (Trap 4c). A gradient's
declaration interpolates `{{background_color_stop.SIZE}}`; `background_color_stop`
is responsive; so the tablet gradient looks up `background_color_stop_tablet`, finds
it empty, throws, and drops the whole tablet rule. The desktop gradient renders
perfectly, which is exactly what makes it so hard to spot.

**Rule: when you set `X_tablet`, set `_tablet` on everything X depends on too** —
every responsive control in its `condition` and in its `needs_value`.

Found by rendering all 25,404 responsive suffixes: 1,433 of them emitted nothing,
every single one a gradient. Handling the breakpoint-matched dependencies took that
to 17.

## Trap 6 — `is_responsive` over-promises, and only rendering knows

The remaining 17 were not a solver bug. They were the schema lying.

`hotspot.width` carries exactly the same args as `container.padding`:

```php
'is_responsive' => true,
'responsive'    => array(),     // empty
// and no width_tablet / width_mobile in the control stack
```

`container.padding_tablet` works. `hotspot.width_tablet` **emits nothing** — verified
in isolation, with only two keys on the element and nothing else to interfere:

```css
/* heading.typography_font_size — the suffix works */
@media(max-width:1024px){ ... font-size:30px; }
@media(max-width:767px) { ... font-size:20px; }

/* hotspot.width — same flag, same shape, no media rule at all */
.elementor-element-d2b2b2b{ --container-width:400px; }
```

So `is_responsive` is a claim, not a guarantee, and an extractor that trusts it
ships a schema that tells you to write a key which silently does nothing.

**The fix is to let the render sweep correct the extractor.** `build-indexes.py`
takes `--verification data/control-verification.csv` and folds the *measured*
result back into the schema:

- `responsive_verified` — the breakpoints whose suffix was seen to emit CSS
- `responsive_broken` — the breakpoints Elementor claims and does not deliver

`el.py` prints `rwd-BROKEN:` on those, and `validate-page.py` makes writing one an
**error**:

```
ERROR  hotspot: `width_tablet` looks legal - Elementor marks `width` responsive -
       but rendering proves it emits no CSS at the tablet breakpoint. The key will
       be stored and ignored.
```

9 controls are in this state on Elementor 4.1.4. Without the sweep, all 9 would
still be in the schema as working responsive controls.

## Trap 7 — CSS is only half of what a control can do

Everything above verifies controls by reading the **compiled stylesheet**. That
method is structurally blind to the other thing an Elementor control can do:

```php
// element-base.php:815 — add_render_attributes()
$this->add_render_attribute( '_wrapper', 'class',
    $controls[ $setting_key ]['prefix_class'] . $setting );
```

**2,573 `(owner, control)` pairs act by putting a class on the element's wrapper**,
not by emitting CSS. `_position`, `hide_tablet`, `hide_mobile`, every `view` /
`shape` / `align` / `position` choose control, the entire transform popover set —
and 1,894 of them have no `css` mapping at all, so a stylesheet sweep cannot even
see that they exist.

They were all shipped in this repo on the strength of *"Elementor registered a
`prefix_class`, so presumably it works"* — the exact reasoning this project exists
to forbid. A schema that verifies one output channel and assumes the other is a
schema that has verified nothing about half its surface.

`sweep-classes.py` renders each one and asserts the class on the wrapper:

```
CLASS-EMITTING CONTROLS   2,573
  verified by class       2,042  (79.4%)   FAILED 0
  host never rendered       523  (20.3%)   the widget produces no markup on a bare page
  skipped, untestable         8  ( 0.3%)
PER-DEVICE CLASS PREFIXES   306   246 verified, 0 failed
classes_dictionary REMAPS    10    10 verified, 0 failed
```

The 523 are not a pass and not a failure: **29 widgets render no markup at all** on
a bare page (`template`, `loop-grid`, `sidebar`, `post-comments`, every
`wp-widget-*`, …), so there is no wrapper for a class to land on. They are flagged
`renders_bare: false` in the schema and `el.py widget <name>` says so, because an
agent that places `loop-grid` and walks away has built an invisible page and been
told nothing. Three of them (`common`, `common-base`, `common-optimized`) are not
placeable widgets at all — they are the registries Elementor injects the shared
controls from.

## Trap 8 — a class control's value is remapped, and its prefix changes per device

Three separate ways the class is not what you would predict from `prefix_class + value`.

**(a) `classes_dictionary` remaps the value first.**

```php
// element-base.php:800
if ( isset( $controls[ $setting_key ]['classes_dictionary'][ $setting ] ) ) {
    $value = $controls[ $setting_key ]['classes_dictionary'][ $setting ];
}
```

`icon-box`'s `position` lists `inline-start | inline-end | block-start | block-end`.
Write the legacy `top` — which is *not* in the option list — and it renders
`elementor-position-block-start`. The dictionary exists so pages saved before
Elementor moved to logical properties keep working, and it means **the option list
is not the list of legal values**. A validator that only knows the options rejects a
page that renders perfectly; this repo's did, until the sweep proved otherwise. All
10 remaps are now verified by rendering them.

**(b) a responsive class control has a different PREFIX per device — there is no
`_tablet` suffix on the class.**

```php
// controls-stack.php:952 — add_responsive_control()
$control_args['prefix_class'] = sprintf( $args['prefix_class'], $device_to_replace );
```

```
'elementor%s-position-'  ->  desktop  elementor-position-
                             tablet   elementor-tablet-position-
                             mobile   elementor-mobile-position-
```

And note *why* these controls are physically duplicated per device rather than
collapsed into the single `is_responsive` form of Trap 2: `add_responsive_control()`
explicitly refuses that optimisation when `prefix_class` is set
(controls-stack.php:909). A collapsing extractor that keeps only the base control's
prefix throws the device prefixes away and leaves the schema quietly claiming the
tablet class is `elementor-position-`. Ours did. `prefix_class_devices` now carries
all of them, and Canary 3 refuses to emit a schema that has lost them.

**(b2) a switcher stores its `return_value`, and that is what gets concatenated.**
Not `true`, and not always `"yes"`. A/B on a live install, same widget, same key:

| what we wrote | class rendered | effect |
|---|---|---|
| `"hide_tablet": "hidden-tablet"` | `elementor-hidden-tablet` | hidden on tablet |
| `"hide_tablet": "yes"` | `elementor-yes` | **nothing at all** |

`"yes"` is the *default* `return_value` for a switcher, which is exactly what makes
this convincing to write and useless to run. `el.py` prints the real one as
`on:'hidden-tablet'`.

**(c) `prefix_class` is not always the prefix of ONE class.** Two of them carry a
space and emit two:

```
'elementor-nav-menu--toggle elementor-nav-menu--' + 'burger'
    ->  elementor-nav-menu--toggle   AND   elementor-nav-menu--burger
```

**(d) zero.** The guard that decides whether a class is emitted at all is:

```php
if ( empty( $setting ) && '0' !== $setting ) { continue; }
```

PHP's `empty()` is true for both `0` and `"0"`, and the escape hatch is a *strict*
comparison against the **string**. So:

| what you write | class emitted |
|---|---|
| `"columns": "0"` | `elementor-grid-0` |
| `"columns": 0` | **nothing at all** |

The editor only ever stores strings, so this failure mode exists purely for people
writing the JSON directly — which is everyone using this skill. `validate-page.py`
errors on it.

## Trap 9 — writing `_elementor_data` leaves a stale rendered-HTML cache

The one that makes a **correct** tree render as the **previous** page.

Elementor stores the HTML it rendered for each element in a post meta
(`Document::CACHE_META_KEY` = `_elementor_element_cache`) and serves it straight back
from `get_builder_content_for_display()`. Its own `Document::save()` deletes the meta.
Writing the meta directly does not.

```php
delete_post_meta( $post_id, \Elementor\Core\Base\Document::CACHE_META_KEY );
```

What it looks like when you have it: the post updates, `_elementor_data` reads back
exactly what you wrote, the compiled CSS file is regenerated and correct — and the
page is unchanged. Nothing errors.

**This repo shipped it.** The entire CSS sweep — 17,421 controls, 0 failures — ran
green against `apply-page.php` with this bug in it, because CSS lives in a separate
file that we deleted and rebuilt every time. It surfaced within a minute of the
first sweep that read the **HTML**: all 14 batches came back byte-identical to
batch 0. The demo page had been serving cached markup too (`html_cache_cleared: true`
on the next apply).

The lesson is not "remember this meta key". It is that **a verifier can only find
bugs in the channel it reads**, and a green run in one channel says nothing at all
about the others.

## Trap 10 — the surface is a property of the INSTALL, not of Elementor

Every trap above is about extracting one machine *correctly*. This one is about the
machine.

The schema shipped for months saying **135 widgets, 3 elements**, extracted from a
live Elementor 4.1.4 / Pro 4.1.2. Extract the *same versions* from a different site
and you get **192 widgets, 13 elements**. Nothing is broken. The extra ones were
never there to find.

```
requires nothing (always present)                     104 widgets
requires plugin:woocommerce                            29     class_exists('woocommerce')
requires a WP legacy widget some plugin registers      33
requires experiment:container                          13
requires experiment:e_atomic_elements  (Elementor V4)  18
requires experiment:nested-elements                     5
```

A module that does not load registers nothing:

```php
// elementor-pro/modules/woocommerce/module.php
public static function is_active() {
    return class_exists( 'woocommerce' );
}
```

So a schema extracted without WooCommerce does not merely *lack*
`woocommerce-product-price` — **it asserts that Elementor has no such widget**, with
total confidence, to anyone who asks. That is not an omission. It is a wrong answer,
and it is the most convincing kind, because every verifier on that machine agrees
with it.

### And third-party plugins inject controls into CORE widgets

Diffing the two sites turned up something worse than missing widgets. Of 130 widgets
present on both, **125 were byte-identical and 5 were not** — and every one of the
five was contaminated:

| widget | difference | who did it |
|---|---|---|
| `accordion`, `nested-accordion` | `rank_math_add_faq_schema` | **Rank Math**, injecting into a core widget |
| `container` | `uc_background_type`, `uc_background_location` | **Unlimited Elements**, injecting into the container |
| `loop-grid`, `loop-carousel` | `product_query_exclude*` | WooCommerce, injecting into Elementor Pro |
| `post-navigation` | `team_taxonomy` vs `product_taxonomy` | **the registered taxonomies**, which differ per site |

The schema was shipping Rank Math's control as if it were Elementor's. Anyone using
it on a site without Rank Math would write a control that does not exist.

### The fix: extract with the other plugins switched off

The same trick as the Free/Pro diff, one axis further. Build the skip-list from the
active plugins and keep only the ones whose surface you actually want:

```bash
ALL=$(wp plugin list --field=name --status=active)
# skip everything except elementor / elementor-pro / woocommerce
wp --skip-plugins="$OTHERS" eval-file tools/extract-elementor-schema.php core+pro > iso-woo.json
```

Three dumps answer three different questions:

| dump | plugins loaded | answers |
|---|---|---|
| `iso-core.json` | elementor | what is FREE |
| `iso-pro.json` | + elementor-pro | what Pro adds (the tier) |
| `iso-woo.json` | + woocommerce | what WooCommerce adds (the `requires`) |

`--skip-plugins` affects only that one CLI process, so all three are safe against a
production site. The result: **zero third-party controls in the schema**, and every
widget carrying what it needs.

### Promo stubs will lie to your tier logic

One more landmine on the way. Elementor core registers a **placeholder for every Pro
widget when Pro is off** — named exactly like the real one, carrying nothing but the
shared Advanced controls, purely to show an upsell:

```php
// elementor/modules/promotions/module.php
public static function is_active() { return ! Utils::has_pro(); }
```

So `woocommerce-product-price` is *present* in the Pro-less dump. Take that at face
value — "it exists without Pro, therefore it is free" — and 26 Pro widgets get
labelled free. They are recognised by their module (`promotions`) and excluded.

### What this costs, stated plainly

The sweeps ran on the site that had no WooCommerce. So of the corrected surface:

```
CSS-driving controls   25,259     22,493 swept (89.0%)     2,766 NOT swept
class-emitting         3,308       3,248 swept (98.2%)
```

The 2,766 are on the WooCommerce and V4 widgets. They are marked unverified in
`data/control-verification.csv` rather than being quietly counted as passes, and the
coverage number went **down** from 99.1% when the schema got more honest. That is the
correct direction for it to move.

## Trap 11 — a rule can be in the stylesheet and LOSE

Every text-level check stops at "the declaration is in the delivered file". Between
that and "the element renders that way" sit specificity, the cascade, and layout —
and none of them leave a trace in any file. So `sweep-browser.py` applies every
batch, opens the PUBLIC url in Chromium, and compares what Elementor DECLARED (read
out of the stylesheet the public downloaded) against what the browser COMPUTED, on
the node the rule actually targets:

```
32,261 computed-style probes across 107 live pages
  verified        6,177   declared == computed, on the right node
  OVERRIDDEN        292   the rule is in the file and the browser did something else
  hover-only      8,135   real rules; not computable without driving the mouse
  no-target-node  8,908   the branch's node does not exist without content/state
  not-comparable  8,745   the browser normalises the value (line-height:1.5 -> px, ...)
```

The 292 are the whole reason to drive a browser, and 242 of them are ONE systemic
fact, adjudicated by Chromium itself listing the matching rules:

```
winner  frontend.min.css   .elementor.elementor .e-con > .elementor-widget
                           { max-width: 100% }                    specificity (0,4,0)
loser   post-1203.css      .elementor-1203 .elementor-element.elementor-element-<id>
                           { max-width: 94px }                    specificity (0,3,0)
```

**Elementor's own base stylesheet outranks its own compiled page CSS.** The
`.elementor.elementor` doubling is deliberate specificity-boosting, and it means the
`max-width` that `_element_width` / `_element_custom_width` compile is dead on every
widget that sits directly inside a container. The width still works - it flows
through the `--container-widget-width` variable instead. Nothing in any file says
the max-width lost; every text check in this repo passed over it for months.

The rest are same-element state overlaps (an `active_color` and a `normal_color`
fighting over the same node when the sweep sets both at once) - an artefact of
testing many controls per element, disclosed in `data/browser-verification.csv`
rather than averaged away.

And three false-failure modes the browser itself taught this tool, each shipped and
then caught by a negative control: `background-size: 950px auto` serialises back as
`950px`; `width`/`height` compute to USED layout values (an 885px image in a 777px
column correctly computes 777px); a `border-width` side computes 0 when that side's
border-style is none.

## Bonus: controls that are dead in the editor, and alive headlessly

`is_control_visible()` also bails when a condition names a control that the widget
does not register:

```php
if ( ! isset( $values[ $pure_condition_key ] ) || null === $values[ $pure_condition_key ] ) {
    return false;
}
```

The button widget's Background group **excludes the image field**, yet
`background_attachment`, `background_repeat`, `background_size` and
`background_position` are all still registered, conditioned on
`background_image[url]!`. There is no `background_image` control on the button, so
in the editor those four can never be switched on. **489 controls across the schema
are in this state** (63 on `mega-menu` alone); they are flagged `dead_dep`.

But `isset()` only asks whether the KEY is in the settings — and we write the
settings. Verified A/B on a live install:

| what we wrote | `background-attachment` | `background-repeat` | `background-size` |
|---|---|---|---|
| what the editor produces | *nothing* | *nothing* | *nothing* |
| `background_image` written raw | `fixed` | `repeat-x` | `contain` |

Put the key the editor cannot create into `settings`, and the dead controls come
alive. It is a small thing, and it is the clearest illustration of what headless
buys you: the editor is *a* client of this data model, not the data model.

## How to know the data is right, rather than hoping

Three verifiers, answering three different questions.

**Does the schema describe your install?**

```bash
python tools/verify-schema.py mine.json --free-dump mine-free.json
```

Walks every `(owner, control)` pair the schema claims and checks it exists on your
site with the same type, and that every Free/Pro claim holds. Exits non-zero if
not, so it can gate a deploy. Against the install it was extracted from:

```
checked 37,964 (owner, control) pairs
Free/Pro claims checked on free widgets/elements: 15,969
FAILURES: 0
```

**Does a page built from the schema actually render what the schema promised?**

```bash
python tools/verify-render.py page.json rendered.css --post-id 9176
```

For every setting, the schema names the CSS properties that control drives (parsed
from Elementor's own `selectors` map). This confirms each one turned up in the
compiled stylesheet, scoped to that element — and that every responsive key landed
inside *that breakpoint's* media query.

```
CSS property assertions: 94/94 passed
```

**Does EVERY control that drives CSS actually work?**

```bash
python tools/sweep-controls.py check sweep/ --out data/control-verification.csv
```

```
DESKTOP     18,853 controls   99.1% covered, 0 failures
RESPONSIVE  25,404 suffixes   96.7% verified by value
```

**Does every control that emits a CLASS actually work?** A different question, read
out of a different artefact — the rendered HTML, not the stylesheet. Nothing above
can answer it.

```bash
python tools/sweep-classes.py check classsweep/ --out data/class-verification.csv
```

```
CLASS-EMITTING CONTROLS   2,573   79.4% verified, 0 failures
PER-DEVICE PREFIXES         306   246 verified
classes_dictionary REMAPS    10    10 verified
```

The render sweep catches Trap 2, because it is the only check that can see a
settings key with no control object behind it. It catches Trap 4, because a hidden
dependency is invisible until the control you set produces nothing. And the class
sweep catches Traps 7-9, because they are all invisible in CSS.

**Does the page the PUBLIC receives have any of this?** None of the checks above
answers that. They all read something from inside the machine: a CSS file off the
server's disk, HTML out of a PHP function call. The theme, the page cache, Varnish
and the CDN sit between all of that and the browser.

```bash
python tools/verify-live.py page.json https://your-site/your-page/
```

```
elements delivered      : 8/8
CSS property assertions : 94 passed
wrapper-class assertions: 17 passed
```

Writing this verifier immediately turned up something none of the others could see:
**a page's styling is split across several Elementor stylesheets.**

```
post-11.css      1,238 bytes    the Kit - global colours and fonts
post-9176.css    5,411 bytes    the page itself
post-47.css      9,009 bytes
post-52.css     18,470 bytes
```

`verify-render.py` reads a single `post-<id>.css` that you hand it, so it is looking
at an incomplete picture by construction — and a property that Elementor chose to
put in the Kit file would read as missing. `verify-live.py` fetches whatever the page
*links*, which is the only complete answer. It also prints the cache headers, because
`x-cache: HIT` on a stale edge is precisely the failure this check exists to catch.

**Both sweeps feed their results back into the schema.** `build-indexes.py` takes
`--verification` and `--class-verification`, and where Elementor's metadata and the
rendered result disagree, **the rendered result wins**:

```bash
python tools/build-indexes.py raw.json --free-dump free.json \
    --verification data/control-verification.csv \
    --class-verification data/class-verification.csv --out data/
```

That is what turns `responsive_broken`, `class_verified` and `renders_bare` into
things the schema *knows* rather than things a human remembers.

> **Ordering rule, learned three times the hard way:** anything measured PER WIDGET
> gets stamped onto controls **after** `compute_common()`, never before. A shared
> control stays in the common set only while it is byte-identical across all 135
> widgets; stamp a per-widget measurement on it first and the set shatters (210
> shared controls collapsed to 192, and the schema grew half a megabyte). It has
> now happened with `tier`, and again with `class_verified`.
