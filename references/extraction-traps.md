# Four traps that make an Elementor schema silently wrong

Everything in `data/` is generated by `tools/extract-elementor-schema.php` running
against a live install. That sounds simple, and the naive version of it is wrong
in four separate ways — each of which produces a schema that *looks* complete,
passes a casual eyeball check, and quietly lies to you.

All four were found the hard way, in this repo, after shipping data that had them.
Three came out of reading Elementor's source; the fourth only surfaced when every
control in the schema was rendered and checked, one at a time
([the sweep](#the-sweep-render-every-control-and-look)). They are written down
because anyone re-extracting for a different Elementor version will hit exactly
the same four.

---

## Trap 1 — WP-CLI looks like the front end, so Elementor hands you a stripped stack

`Elementor\Core\Frontend\Performance::should_optimize_controls()`:

```php
public static function should_optimize_controls() {
    if ( null === static::$is_frontend ) {
        static::$is_frontend = (
            ! is_admin()
            && ! Plugin::$instance->preview->is_preview_mode()
            && ! defined( 'REST_REQUEST' )
        );
    }
    return static::$is_frontend;
}
```

A `wp eval-file` run is not admin, not preview, and not REST. All three
conditions pass, so Elementor concludes it is rendering the front end and
switches to its lean control path — the one built for speed, not for
introspection. Two things then happen:

| Where | What it does |
|---|---|
| `includes/base/controls-stack.php` | UI controls (heading, divider, raw_html, button, alert, notice) are cut down to `['type', 'section']` — labels gone |
| `includes/managers/controls.php` | Every control with `selectors` (i.e. every *style* control) is diverted into a separate `style_controls` stack, and section markers lose their `label` and their real `tab`, defaulting to `'content'` |

Measured on Elementor 4.1.4:

```
                        optimised    full     lost
container controls            310     458    32%
all settable controls      20,485  37,964    46%
controls carrying a label       7  36,753    ~100%
tab attribution           all 'content'  layout/style/advanced   broken
```

**Nearly half the control surface, and every scrap of tab and label metadata,
vanishes — with no error.** A schema built this way will tell you the Heading
widget has no style controls.

The fix is to turn the flag off before anything touches an element:

```php
$rp = new ReflectionProperty( \Elementor\Core\Frontend\Performance::class, 'is_frontend' );
$rp->setAccessible( true );
$rp->setValue( null, false );
```

Order matters: `$is_frontend` is a cached static, and control stacks are built
lazily on first `get_controls()`. Flip it first, before you read anything.

Two consequences to know about:

- With optimisation off, Elementor `wp_die()`s on any addon that registers a
  control outside a section (`controls-stack.php` only tolerates that in the
  optimised path). One sloppy third-party plugin kills the whole extraction. The
  extractor therefore decides a widget's tier *before* calling `get_controls()`
  on it, and never builds controls for widgets it is going to discard.
- Labels are translated at registration time, so the dump inherits the site's
  locale. `switch_to_locale( 'en_US' )` before extracting, or you ship a schema
  full of `'背景'` where the world expects `'Background'`.

The extractor asserts its way out of this rather than trusting it worked: it
checks that `container` / `background_color` reports `tab = 'style'`, and exits
non-zero if not.

## Trap 2 — responsive is two mechanisms, and the obvious test only finds one

Half the point of the schema is answering "can I write `padding_tablet` here?".
Getting that wrong in the *rejecting* direction is worse than not knowing.

`add_responsive_control()` does not do one thing. It does two, and they look
nothing alike in the control stack:

**Mechanism A — the common one.** The control is registered **once**. There is no
`padding_tablet` entry in the stack at all; the breakpoint variants are resolved
at render time by looking up `"{control}_{device}"` in the saved settings. The
raw args of the container's `padding`:

```php
'type'          => 'dimensions',
'responsive'    => array(),      // EMPTY
'is_responsive' => true,         // <- the actual flag
```

**Mechanism B.** Real suffixed siblings exist in the stack alongside the base:

```php
// sticky_offset
'responsive' => array( 'max' => 'desktop' ),   // non-empty
// and sticky_offset_tablet, sticky_offset_mobile are separate controls
```

Testing `! empty( $ctrl['responsive'] )` — the obvious test — catches **only
mechanism B**. On Elementor 4.1.4 that is 58 controls on the container while
mechanism A covers the ones everybody actually uses: padding, margin, width,
font size, gap.

Before the fix, 9.8% of controls were marked responsive. After: **30.1%**. The
schema had been rejecting `padding_tablet` as an unknown control.

The authoritative flag is **`is_responsive`**. A control is responsive if
`is_responsive` is true *or* it has suffixed siblings; the usable suffixes are
the site's **active** breakpoints (`el.py breakpoints`).

The extractor has a second canary for this: if `container` / `padding` does not
come out responsive, it refuses to emit.

**This is also why `verify-render.py` exists.** A control name you can look up in
a stack can be checked against that stack. A settings key with *no control object
behind it anywhere* cannot — the only way to know `padding_tablet` works is to
write it, let Elementor compile the CSS, and confirm a `@media(max-width:1024px)`
rule came out the other end. Which it does:

```css
@media(max-width:1024px){ .elementor-element-1a2b3c4{ --padding-top:56px; ... } }
```

## Trap 3 — a control's tier is not its widget's tier

You can read a *widget's* tier off the filesystem: whichever plugin directory
defines the class. You cannot do that for a *control*, because **Elementor Pro
reaches into the free widgets and injects controls into them.**

Open the free Heading widget on a site with Pro and you will find Motion Effects,
Sticky, Custom CSS, Display Conditions and Custom Attributes sitting in its
Advanced tab. Inherit the owner's tier and all of them get labelled `free`. Build
on that assumption and the page renders perfectly on your machine and silently
loses its styling on a Free install. No error. No warning.

Measuring it is easy once you stop trying to reason about it:

```bash
wp eval-file tools/extract-elementor-schema.php core+pro > pro.json
wp --skip-plugins=elementor-pro eval-file tools/extract-elementor-schema.php core+pro > free.json
python tools/build-indexes.py pro.json --free-dump free.json --out data/
```

`--skip-plugins` affects **only that one CLI process**. Nothing is deactivated;
this is safe to run against production.

Diff the two and the boundary falls out — measured, not argued:

| | Free 4.1.4 | + Pro 4.1.2 |
|---|---|---|
| controls on every widget | 165 | **211** (+46) |
| controls on `container` | 277 | **356** (+79) |
| control types | 52 | **59** (+8, −1) |
| group controls | 11 | **16** (+5) |

The 46 injected into **every** widget:

| Section | n | What |
|---|---|---|
| `section_effects` | 43 | all `motion_fx_*` (37) + `sticky*` (6) |
| `section_custom_css` | 1 | `custom_css` |
| `_section_attributes` | 1 | `_attributes` — custom HTML attributes. **Pro.** Most people assume free |
| `_section_style` | 1 | `e_display_conditions` |

Pro-only group controls: `motion_fx`, `posts`, `query-group`, `related-query`,
`taxonomy-query`.
Pro-only control types: `query`, `template_query`, `conditions_repeater`,
`form-fields-repeater`, `fields_map`, `media-preview`,
`animation_menu_dropdown`, `animation_slides_content`.

### The counting quirk you will hit, and what it means

The Pro-less build reports **more** widgets than the Pro build (145 vs 135), and
the extras are all WooCommerce ones. That is not a bug in the diff.

When Pro is inactive, **Elementor core registers stub "upgrade to Pro" widgets**
so the editor can advertise them. They carry exactly the shared common controls
and nothing of their own, and they are identifiable by a control type that exists
*only* when Pro is absent: **`promotion_control`**
(`Elementor\Modules\Promotions\Controls\Promotion_Control`).

So a widget merely *existing* in the Pro-less build does not make it free. The
tier logic recognises the stubs (zero controls of their own) and does not treat
their controls as free. Once that is accounted for, the books balance exactly:

```
52 free control types  −  1 (promotion_control, free-only)  +  8 (Pro-only)  =  59
```

and the two independent methods — tier-by-filepath and tier-by-presence — agree on
every user-facing widget. The only five they disagree on (`common`, `common-base`,
`common-optimized`, `e-component`, `inner-section`) are Elementor's internal base
classes, which have no controls of their own and are not widgets anyone places.

### One more piece of semantics

A control's tier answers: *"assuming I can use this widget at all, does **this
control** additionally require Pro?"* On a widget that is itself Pro the question
is circular, so per-control tier there carries no information and is not checked.
What matters — and what is checked, on all 15,969 of them — is a Pro-only control
sitting on a **free** widget.

## Trap 4 — a control has THREE ways of being gated, and `condition` is only one

This is the one you cannot find by reading the schema. It only shows up when you
set a control, render the page, and notice nothing happened.

A control emits its CSS only if **all three** of these hold. Handle one or two of
them and a large fraction of Elementor looks broken when it is merely gated.

### 4a. `condition` — the simple form (7,946 controls)

A flat map, checked with equality or membership. Three syntaxes, all of which
appear in real controls:

```jsonc
"background_background": ["classic", "gradient"]   // must be one of
"typography_typography!": ""                       // trailing ! = must NOT equal
"selected_icon[value]!": ""                        // index into the value object
```

A negated condition can carry a **list**, and that list is a set of values to
*avoid* — not a single one:

```jsonc
"_border_color": { "condition": { "_border_border!": ["", "none"] } }
```

`_border_border` must be neither `""` **nor `"none"`**. "Pick the first non-empty
option" lands you on `"none"`, which is precisely the value that switches the
border off — so the colour and the width never emit and the control looks broken
when it is not. This cost 3,123 false failures in the first sweep run.

### 4b. `conditions` — the advanced form (152 controls)

A completely separate mechanism with its own syntax: a boolean tree with `relation`
(and/or), nested `terms`, and comparison operators.

```jsonc
// text-editor.column_gap
{ "relation": "or",
  "terms": [ { "name": "text_columns", "operator": ">",   "value": 1 },
             { "name": "text_columns", "operator": "===", "value": "" } ] }
```

The operators are exactly Elementor's (`includes/conditions.php::compare`):
`==`, `!=`, `!==`, `in`, `!in`, `contains`, `!contains`, `<`, `<=`, `>`, `>=`, and
`===` by default.

**Controls gated this way have an empty `condition`.** Check only `condition` and
they read as unconditional — so you set one, nothing happens, and there is nothing
in the schema to explain why. 152 controls, including `counter.number_gap`,
`off-canvas.offcanvas_animation_duration` and every `motion_fx` transform-origin.

### 4c. `needs_value` — not a condition at all

The killer, because no condition is unmet and nothing looks wrong.

Elementor's CSS generator (`core/files/css/base.php`) expands every `{{...}}`
placeholder in a declaration, and:

```php
if ( '' === $parsed_value ) {
    ...
    throw new \Exception();
}
} catch ( \Exception $e ) {
    return;                 // <- the entire rule is abandoned
}
```

So a declaration that interpolates **another control's** value emits nothing when
that other control is empty. The gradient is the classic:

```
background-image: linear-gradient({{SIZE}}{{UNIT}},
    {{background_color.VALUE}} {{background_color_stop.SIZE}}...,
    {{background_color_b.VALUE}} ...)
```

Set `background_background: "gradient"` and `background_gradient_angle`, satisfy
every documented condition, and you get **no gradient** — because
`background_color` and `background_color_b` are empty. There is no error. There is
nothing in `condition` to warn you.

**499 controls (4.9%) have one of these hidden cross-control dependencies.** The
extractor now parses them out of the `selectors` map and stores them as
`needs_value`; `el.py` prints them as `needs-value-of:`; and `validate-page.py`
makes it an error rather than letting you ship a gradient that does nothing.

## The sweep: render every control and look

`verify-render.py` checks a page you wrote. That only ever covers the controls you
happened to use — 94 of them, on the demo page. It says nothing about the other
16,000.

`tools/sweep-controls.py` closes that gap. For **every** control in the schema that
claims to drive CSS, it synthesises a legal value, solves the dependency chain
needed to make the control take effect (all three mechanisms above), writes the
pages, reads back the stylesheet Elementor compiled, and asserts the value came out.

Each control gets a value **unique to it** — a distinct hex colour, a distinct
pixel size — so "did `title_color` work?" is answered by finding that exact colour
in the CSS, not by noticing that *something* set a `color`. Property-presence would
pass on a page where a different control happened to write the same property.

```bash
python tools/sweep-controls.py plan --out sweep/ --post-id <a draft post>
# apply each batch, capture post-<id>.css  (sweep/RUN.sh shows the loop)
python tools/sweep-controls.py check sweep/ --out data/control-verification.csv
```

Result on Elementor 4.1.4 / Pro 4.1.2 — 550 element variants across 55 pages:

```
controls asserted     16,778
  verified by value   13,586  (81.0%)   the exact value we wrote is in the CSS
  property only        3,192  (19.0%)   right property, value not literally assertable
  FAILED                   0  ( 0.0%)
```

Per-control results ship in **`data/control-verification.csv`**, so "is this control
actually known to work?" is a lookup rather than a hope.

### What the sweep found

The first run failed on **3,123 controls (17.2%)**. Every one of them was a bug —
in the sweep's own solver, or in the schema, or in Elementor:

| Failures | Cause | Whose bug |
|---|---|---|
| ~1,300 | negated conditions with a list (`_border_border!: ['', 'none']`) satisfied with `'none'` | the solver's |
| ~900 | media controls given an empty url, so `url("{{URL}}")` interpolated to nothing and the rule was dropped | the solver's |
| ~800 | gradient/background controls whose CSS interpolates sibling controls that were never set | **the schema's** — it had no `needs_value` |
| 6 | controls gated only by the advanced `conditions` form | **the schema's** — nothing read it |
| 1 | `counter.number_gap` — see below | **the tooling's**, and an Elementor defect |

After fixing all of them: **0 failures**.

### An actual Elementor defect, found by looking

The last failure was `counter.number_gap`. Its conditions were satisfied, its value
was written, and no `gap` appeared. It turned out `gap` **was** emitted — into a
rule that Elementor had already corrupted:

```css
.elementor-element-0994e53 .elementor-counter-number-wrapper{
    text-align:{{VALUE}};        /* a literal, unexpanded placeholder */
    --counter-suffix-grow:1;
    gap:245px;                   /* our value, right there */
}
```

**Elementor 4.1.4 writes a literal `{{VALUE}}` into the compiled stylesheet** on the
`counter` widget. That is invalid CSS, and the stray braces stop any `[^{}]*`-style
parser mid-rule — which is why the sweep reported a false failure that looked
exactly like a schema bug. Both `sweep-controls.py` and `verify-render.py` now
count braces instead.

The lesson generalises past this one widget: **do not parse Elementor's compiled
CSS with a regex that assumes it is well-formed.** It is not, always.

## How to know the data is right, rather than hoping

Three verifiers, answering three different questions.

**Does the schema describe your install?**

```bash
python tools/verify-schema.py mine.json --free-dump mine-free.json
```

Walks every `(owner, control)` pair the schema claims and checks it exists on your
site with the same type, and that every Free/Pro claim holds. Exits non-zero if
not, so it can gate a deploy. Against the install it was extracted from:

```
checked 37,964 (owner, control) pairs
Free/Pro claims checked on free widgets/elements: 15,969
FAILURES: 0
```

**Does a page built from the schema actually render what the schema promised?**

```bash
python tools/verify-render.py page.json rendered.css --post-id 9176
```

For every setting, the schema names the CSS properties that control drives (parsed
from Elementor's own `selectors` map). This confirms each one turned up in the
compiled stylesheet, scoped to that element — and that every responsive key landed
inside *that breakpoint's* media query.

```
CSS property assertions: 94/94 passed
```

**Does EVERY control in the schema actually work?**

```bash
python tools/sweep-controls.py check sweep/ --out data/control-verification.csv
```

```
controls asserted     16,778
  verified by value   13,586  (81.0%)
  FAILED                   0
```

The second catches Trap 2, because it is the only check that can see a settings key
with no control object behind it. The third catches Trap 4, because a hidden
dependency is invisible until the control you set produces nothing.
